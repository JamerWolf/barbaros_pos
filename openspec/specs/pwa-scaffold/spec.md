# PWA Scaffold Specification

## Purpose

Define the minimum Progressive Web App (PWA) configuration for the Bárbaro's POS web client so that tablets can install the application, cache static assets, and continue working when the local Wi-Fi connection drops.

## Requirements

### Requirement: Web App Manifest

The `apps/web` workspace SHALL contain a valid Web App Manifest that provides the application name, short name, start URL, display mode, and icons.

#### Scenario: Happy path — manifest is discoverable

- GIVEN `apps/web/public/manifest.json` exists with valid PWA fields
- WHEN the browser requests the manifest
- THEN it SHALL be served at the root path `/manifest.json`

#### Scenario: Edge case — manifest is missing

- GIVEN the manifest file is absent
- WHEN the browser requests `/manifest.json`
- THEN the server SHALL respond with a 404 status
- AND the browser SHALL NOT offer the install prompt

### Requirement: Service worker registration

The web client SHALL register a service worker generated by Workbox during the production build.

#### Scenario: Happy path — service worker registered

- GIVEN the production build has generated a service worker
- WHEN the application loads in a supported browser
- THEN the client code SHALL call `navigator.serviceWorker.register('/sw.js')`
- AND the browser SHALL install the service worker

#### Scenario: Edge case — service worker fails to register

- GIVEN the browser does not support service workers
- WHEN the registration code runs
- THEN the application SHALL continue working as a normal web app
- AND it SHALL log a warning instead of crashing

### Requirement: Precache of static assets

The service worker SHALL precache the application shell (HTML, JS, CSS) so that the app loads even without network access.

#### Scenario: Happy path — offline load

- GIVEN the service worker has been installed and the app shell is cached
- WHEN the device has no network connection
- THEN the browser SHALL serve the cached `index.html` and assets
- AND the application SHALL render the login screen

#### Scenario: Edge case — first load without network

- GIVEN the app has never been loaded before
- WHEN the device has no network connection
- THEN the browser SHALL show its default offline page
- AND the application SHALL NOT render because no assets exist in cache

### Requirement: Build-time Workbox integration

The `apps/web` Vite configuration SHALL integrate Workbox to generate the service worker and precache manifest as part of the production build.

#### Scenario: Happy path — build generates service worker

- GIVEN the Workbox plugin is configured in `vite.config.ts`
- WHEN the developer runs the production build
- THEN a service worker file SHALL be generated in the output directory
- AND the build SHALL complete without errors

#### Scenario: Edge case — Workbox plugin missing

- GIVEN the Workbox plugin is not configured
- WHEN the production build runs
- THEN no service worker SHALL be generated
- AND the application SHALL NOT be installable as a PWA

### Requirement: Development-time behavior

The service worker SHALL NOT interfere with Vite's Hot Module Replacement during development.

#### Scenario: Happy path — development runs without service worker

- GIVEN `npm run dev:web` is running
- WHEN a developer edits a source file
- THEN Vite HMR SHALL update the browser immediately
- AND no stale cached assets SHALL be served

## Notes

- This is the base PWA scaffold only. Advanced offline strategies (background sync, queued mutations, IndexedDB caching) are out of scope for this change and will be added in later changes related to accounts and payments.
- The production build target is the only scenario where Workbox runs. Development deliberately skips service worker registration.
